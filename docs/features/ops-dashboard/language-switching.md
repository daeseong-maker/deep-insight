# Deep Insight Ops — Admin Dashboard i18n

> Internationalization (Korean/English) for the admin dashboard pages.

**Last Updated**: 2026-03

---

## Key Terms

| Term | What It Is |
|------|-----------:|
| **i18n** | Short for "internationalization" (i + 18 letters + n). The practice of designing software so it can be adapted to different languages without code changes. Instead of hardcoding text like `"Login"`, you use a key like `login_btn` that maps to `"Login"` in English and `"로그인"` in Korean. |
| **Translation key** | A string identifier (e.g., `login_btn`) that maps to a human-readable string in each language. Keys are language-neutral; their values change per language. |
| **`data-i18n` attribute** | A custom HTML attribute that marks an element for automatic text translation. When the language changes, JavaScript scans all elements with `data-i18n` and replaces their text content with the translated value. |
| **`localStorage`** | A browser API that stores key-value pairs persistently (survives page reload and navigation). Used here to remember the user's language choice across admin pages. Unlike cookies, localStorage data is never sent to the server. |
| **DOM (Document Object Model)** | The browser's in-memory representation of the HTML page as a tree of objects. JavaScript manipulates the DOM to update text, add elements, or change styles — all without reloading the page. |

## Architecture

```
ops/static/admin-i18n.js     — Translation dictionary + language toggle logic
ops/static/login.html        — Login page (uses i18n)
ops/static/jobs.html         — Job list dashboard (uses i18n)
ops/static/job.html          — Job detail page (uses i18n)
ops/admin_router.py          — Serves JS files via /admin/static/{filename}
```

### How It Works

```
Page loads
  │
  ▼
Browser loads admin-i18n.js
  │  Contains: adminI18n dict (en + ko), t(), toggleAdminLang(), applyAdminLang()
  │  Reads: localStorage.getItem('adminLang') → defaults to 'ko'
  ▼
applyAdminLang() runs
  │
  ├─ Static text: scan all elements with data-i18n attribute
  │    <th data-i18n="th_status">Status</th>
  │    → element.textContent = adminI18n['ko']['th_status']
  │    → "상태"
  │
  ├─ Placeholders: scan all elements with data-i18n-placeholder attribute
  │    <input data-i18n-placeholder="login_password_ph" placeholder="Enter password">
  │    → element.placeholder = adminI18n['ko']['login_password_ph']
  │    → "비밀번호를 입력하세요"
  │
  └─ Dynamic content: call onLangChange() if defined by the page
       → re-renders table rows or detail rows using t() function
```

## Two Types of Translated Text

### 1. Static Text (data-i18n attributes)

Text that exists in the HTML from the start — labels, headers, buttons.

```html
<!-- HTML element with data-i18n attribute -->
<th data-i18n="th_status">Status</th>

<!-- applyAdminLang() finds this element and updates it -->
<!-- In Korean: <th data-i18n="th_status">상태</th> -->
<!-- In English: <th data-i18n="th_status">Status</th> -->
```

The initial text content (`Status`) serves as the English default and is visible before the JS loads.

### 2. Dynamic Text (t() function)

Text generated by JavaScript when rendering data from the API — table rows, detail labels, error messages.

```javascript
// Instead of hardcoded string:
statusSpan.textContent = 'Start';

// Use the t() function:
statusSpan.textContent = t('status_start');
// → 'Start' (English) or '시작' (Korean)
```

The `t()` function looks up the key in the current language dict, falls back to English, then falls back to the raw key:

```
t('status_start')
  → adminI18n['ko']['status_start']  →  '시작'     ✓ found
  → adminI18n['en']['status_start']  →  (fallback)
  → 'status_start'                   →  (last resort)
```

## Language Toggle Flow

```
User clicks EN/KR button
  │
  ▼
toggleAdminLang()
  │  Flip: adminLang = 'ko' ↔ 'en'
  │  Save: localStorage.setItem('adminLang', 'en')
  ▼
applyAdminLang()
  │
  ├─ Update toggle button text: 'EN' ↔ 'KR'
  ├─ Update all data-i18n elements (static text)
  ├─ Update all data-i18n-placeholder elements (input placeholders)
  └─ Call onLangChange() → re-render dynamic content
```

### Why localStorage?

The admin dashboard has 3 separate HTML pages (login, jobs, job detail). When the user navigates between them, a new page loads and all JavaScript state is lost.

Without localStorage:
```
jobs.html (Korean) → click job → job.html loads → resets to Korean (default)
                                                   ↑ user's English choice lost
```

With localStorage:
```
jobs.html (switch to English) → localStorage saves 'en'
  → click job → job.html loads → reads 'en' from localStorage → English
```

## Page-Specific Re-rendering (onLangChange)

Each page optionally defines an `onLangChange()` function that `applyAdminLang()` calls after updating static elements:

| Page | onLangChange | What It Does |
|------|-------------|--------------|
| `login.html` | Not defined | No dynamic content to re-render (all labels are static `data-i18n`) |
| `jobs.html` | `renderJobs()` | Re-renders the entire table body with translated status badges, "running...", "STALE", and empty state text |
| `job.html` | `renderJob(currentJob)` | Re-renders all detail rows with translated labels ("Started" → "시작 시각") |

## Serving the JS File

The `admin-i18n.js` file is served by a dedicated route in `admin_router.py`:

```
GET /admin/static/admin-i18n.js
  │
  ▼
admin_static() route handler
  │  ✓ Extension check: only .js and .css allowed (not .html)
  │  ✓ File exists check
  │  ✓ Path traversal check: resolved path must be inside ops/static/
  ▼
FileResponse(file_path)
```

**Why not serve HTML files through this route?**

The HTML pages (jobs.html, job.html) are served through auth-protected routes (`/admin/dashboard`, `/admin/dashboard/{job_id}`) that require a valid JWT cookie. Serving them via `/admin/static/` would bypass authentication. Although the HTML files are just templates (no sensitive data — the APIs are separately protected), restricting the static route to `.js` and `.css` follows the principle of least privilege.

## How to Add a New Translation

### Step 1: Add the key to admin-i18n.js

Add the key to both `en` and `ko` sections:

```javascript
const adminI18n = {
    en: {
        // ... existing keys ...
        my_new_label: 'New Label',
    },
    ko: {
        // ... existing keys ...
        my_new_label: '새 라벨',
    }
};
```

### Step 2a: For static text — add data-i18n to the HTML element

```html
<span data-i18n="my_new_label">New Label</span>
```

### Step 2b: For dynamic text — use t() in JavaScript

```javascript
var label = t('my_new_label');
element.textContent = label;
```

### Step 2c: For input placeholders — add data-i18n-placeholder

```html
<input placeholder="New Label" data-i18n-placeholder="my_new_label">
```

## Translation Key Naming Convention

Keys follow a prefix-based naming pattern:

| Prefix | Used In | Example |
|--------|---------|---------|
| `admin_` | Shared header elements | `admin_title`, `admin_logout` |
| `login_` | Login form | `login_btn`, `login_err_empty` |
| `change_` | Change password form | `change_title`, `change_btn` |
| `filter_` | Job list filter buttons | `filter_all`, `filter_start` |
| `th_` | Table column headers | `th_status`, `th_query` |
| `status_` | Status badge display text | `status_start`, `status_success` |
| `jobs_` | Job list page specific | `jobs_empty`, `jobs_running` |
| `detail_` | Job detail page specific | `detail_title`, `detail_loading` |
| `label_` | Job detail row labels | `label_job_id`, `label_duration` |

## Comparison with Main Web UI i18n

| Aspect | Main Web UI | Admin Dashboard |
|--------|------------|-----------------|
| Translation file | `static/js/i18n.js` | `ops/static/admin-i18n.js` |
| Variable name | `translations` | `adminI18n` |
| Language variable | `currentLang` | `adminLang` |
| Persistence | None (resets on reload) | `localStorage` |
| Default language | Korean | Korean |
| Toggle function | `toggleLanguage()` | `toggleAdminLang()` |
| Pages | 1 (SPA) | 3 (login, jobs, job detail) |
| Dynamic re-render | `renderPromptExamples()` etc. | `onLangChange()` pattern |

Different variable names (`adminI18n` vs `translations`) prevent conflicts if both systems are ever loaded on the same page.
